# Cable Handoff

Branch: `wiring`

This document summarizes the current cable work so a later session can continue without reconstructing the discussion.

## Context

The goal is to introduce `Cable` next to `Bus` and `Route`.

Current conceptual model:

- `Route` is a two-endpoint local leg.
- `Bus` is a named/keyed trunk. It selects connected pins/pads by name or alias.
- `Cable` is an endpoint-owned trunk. It can carry multiple connections and can connect to a bus through a bridge.
- Internally, route connection strings still use `BUS` for both bus and cable trunk-like endpoints. This is acceptable for now because bus and cable behavior are getting closer at the route matrix level. A future neutral internal name would be `TRUNK`, not separate `BUS` and `CBL`.
- Generated apps still resolve to explicit pin target tables. Cable and bus topology are editor/model constructs, not runtime primitives.

Important design decisions:

- `Cable` and `Bus` remain separate trunk owners because their UI behavior and lifecycle differ.
- Tacks are unified again: `BusTack` is now the single tack object used by both buses and cables.
- A cable has no `name`; only buses have names.
- Tacks no longer use arrows to repeat message flow.
- Tack visuals now indicate selectivity: non-selective tacks render as bullets, selective receiver tacks render as bullets with an outline.
- Selectivity belongs to the receiving endpoint connection, not to the cable or bus trunk itself.
- Cable endpoint tacks are still marked with `is.endpoint`.
- Cable/bus bridge tacks are still marked with `is.bridge`.
- A cable-to-bus bridge is represented logically by two tacks at the same point: one cable tack and one bus tack.
- Visually, the bridge should be read as a shared trunk connection, not as a message-flow arrow.
- Bus-to-bus and cable-to-cable bridge routes are not allowed.
- Cable-to-bus and bus-to-cable bridge routes are allowed.
- There can only be one bridge path between a given cable and bus. Duplicate/circular bridge patterns are rejected.

## Cable

`Cable` is now standalone and no longer derives from `Bus`.

Current shape:

- `is.cable = true`
- Owns `uid`, `wire`, `tacks`, and visual state.
- Does not own `name`.
- `makeRaw()` emits only `start` and `wire`.
- `cook(raw)` restores wire.
- `render(ctx)` draws cable styling and cable tacks, without bus labels.
- `isKeyed()` is retained only as a compatibility stub and currently returns false. Selectivity is handled by tacks/endpoints.
- `moveSegment(segment, delta)` refuses to move a cable segment that has an endpoint tack.
- Cable segment movement uses cable-specific adjacent-tack blocking. Cable tacks sit on the cable line, so the bus limit logic was too strict/loose in different cases. A segment may move away from a neighboring tack if it starts inside the tack zone, but cannot move through the tack to the other side.
- `resumeDrawing(segment, point)` supports shift-click redraw of a cable from a segment point.
- `connectEndpoint(widget)` creates a new endpoint tack/route leg and snaps the cable endpoint to the pin/pad center.
- `collapseIfOnlyEndpointTacks(node)` can collapse a cable back into a route when only two endpoint tacks remain.

Copied/adapted bus-like behavior exists for:

- hit testing
- selection and movement
- adding/removing tacks
- route creation
- copy/restore of wire and tack wires
- source/target/topology helpers

## Unified Tack

`BusTack` is now the single tack class for both buses and cables.

Current shape:

- `is.tack = true`
- `is.endpoint = false`
- `is.bridge = false`
- Stores owner trunk in `bus`, which may be a real bus or a cable.
- Has no message-flow arrow orientation.

Rendering:

- Endpoint tacks are currently hidden.
- Non-selective tacks render as bullets.
- Selective receiver tacks render as bullets with an outline.

Connection behavior:

- Output tacks publish onto the owner trunk.
- Input tacks decide whether to accept all compatible messages or only matching names/aliases.
- Current default: input connections on buses are selective; input connections on cables are non-selective.
- Bridge traversal continues through connected bus/cable tacks.
- `makeConxList()` uses visited sets and blocked route arguments to avoid bridge traversal loops.

## BusTack Bridge Behavior

Bridge behavior:

- `is.bridge` was added.
- Bridge tacks snap/place on the trunk segment using tack-to-tack route endpoints.
- Bridge movement reroutes the bridge route from current tack centers instead of invoking arrow orientation.

## Route Connections

`route-connect.js` still classifies bus and cable trunk endpoints as `BUS` in `conxString()`.

Implemented bridge rules:

- `BUS-BUS` now means a potential trunk-to-trunk bridge.
- `canBridge(from, to)` allows only cable-tack to bus and bus-tack to cable.
- Bridge graph traversal rejects duplicate/circular cable-bus paths.
- `connect('BUS-BUS')` creates the target tack, marks the bridge via tack `setRoute()`, calls `autoRoute()`, then updates rxtx tables.
- `disconnect('BUS-BUS')` disconnects rxtx bridge effects and removes the route.

Implemented route promotion while drawing:

- `Route` now has `is.route`, `is.hoverOk`, and `is.hoverNok`.
- While drawing a route, hovering over a complete direct route highlights it as a valid/invalid promotable target.
- On mouse-up over a valid route, `routeDrawToRoute` converts the target route to a cable and connects the drawn route to that new cable in one undoable edit.
- Valid route targets currently must be complete direct routes, not tack routes.
- Pin/pad-to-route is allowed unless the pin/pad is already an endpoint of that route.
- Bus-tack-to-route is allowed.
- Cable-tack-to-route is rejected for now to avoid cable-to-cable connections.

Route clone/reconnect behavior was updated so tack-origin routes can be restored for undo/redo.

## Route Drawing And Moving

Several fixes were made for routes that terminate on a horizontal trunk segment.

Current behavior:

- A route from a horizontal cable/bus segment to a pin/pad should be a three-point route with two segments.
- The vertical segment should sit at the trunk tack.
- Moving that vertical segment slides the tack along the cable/bus segment.
- Bridge routes between cable and bus now call `autoRoute()` after both tacks exist.
- `BUS-BUS` autorouting now uses the same horizontal-trunk three-point route rule.
- `route-moving.js` now handles tack-to-tack routes by checking both endpoints for horizontal trunk tacks before using generic HH/HV/VH adjustment.

Known detail:

- If neither bridge endpoint is on a horizontal trunk, bridge routes currently fall back to four/six point autorouting.

## Endpoint Behavior

Cable endpoints are represented as cable tacks marked `is.endpoint`.

Current behavior:

- Cable endpoint segment cannot slide independently.
- When the pin/pad endpoint moves, the cable endpoint tack and cable trunk endpoint move with it.
- When a pin moves inside a node, the cable endpoint remains attached by route adjustment.
- When cable redraw finishes on a pin or pad, the cable endpoint snaps to the widget center, not to the mouse release point.
- Endpoint tacks are hidden visually.

This is the chosen data shape for now: endpoint tacks are structural cable tacks rather than separate `from` / `to` properties on cable.

## Cable Redraw

Shift-click on a cable segment now starts a redraw from that point, similar to shift-click route redraw.

Current behavior:

- The cable keeps the side closest to one existing cable endpoint and discards the other side.
- If the start point is closer to the cable's first endpoint, the cable is reversed before trimming so redraw always continues from the retained tail.
- Tacks on the discarded part are disconnected and removed before redraw continues.
- On mouse-up over a pin or pad, a new endpoint tack and route leg are created.
- On mouse-up without a valid endpoint, the cable and all remaining tacks are deleted.
- Undo/redo for this behavior is handled by `cableDraw`, which restores both cable wire and tack set.

Notes:

- Cable redraw currently uses the normal `mouseHit()` result for detecting the endpoint on release.
- A pin is detected only when the release point is inside the pin rectangle. If the visual feels too strict, inspect the pin hit rectangle rather than adding a special cable-only hit path.

## Collapse Behavior

Ctrl-clicking a route promotes it to a cable.

Collapse back to a route is partially implemented:

- If a cable has only two endpoint tacks left, it can collapse back into a normal route.
- `redox-route.js` uses a `checkForCollapse(route, node)` helper to snapshot possible collapse state.
- Redox actions perform `route.disconnect()` or `route.popFromRoute()` themselves after calling the helper.
- `view.root` is passed directly as the node that owns the cable array.
- Canceling an abandoned route after ctrl-click conversion should also trigger collapse if only endpoint tacks remain.

This has been manually tested in the abandoned route case, but undo/redo and edge cases still need more testing.

## Serialization And Schema

Schema:

- `cli/context/0.9.6/viz.schema.json` includes `cables`.
- Cable raw objects do not include `name`.

Route raw endpoints:

- Cable endpoints serialize with `(cable N ...)`.
- Endpoint and bridge flags are serialized when present, for example `(cable 2 endpoint bridge)`.
- Raw parsing restores `cable`, `index`, `endpoint`, and `bridge`.

`node-group-json.js`:

- Assigns temporary `_rawIndex` values to cables during serialization.
- Restores cables and their tacks during cook.
- Restores `is.endpoint` and `is.bridge`.
- Marks both ends as bridge when a route connects two tacks.

Test coverage:

- `core/tests/cable-route-json.test.js` covers unnamed cable endpoint parsing and flags.

## RX/TX Traversal

The current direction is graph traversal:

- For every pin/pad, find all ultimately connected compatible pins/pads.
- Continue traversal through routes, cable tacks, buses, and bridge tacks.
- Deduplicate targets.
- Avoid cycles with visited sets.

Cable/bus bridge semantics:

- A cable connected to a bus does not itself become keyed.
- Messages from cable outputs can traverse the bridge onto the bus.
- Bus-side input tacks remain selective, so bus input `A` only accepts source `A` unless aliases/names say otherwise.

Open concern:

- Grouping input pins on a cable and then bridging that cable to a bus is governed by the receiver-side tack rules, not by the cable trunk type.

## Known Caveats

- Internal connection naming still uses `BUS` for both bus and cable trunk endpoints.
- `tack.bus` is now the intentional owner-trunk reference and may point to either a bus or a cable.
- Bridge rxtx traversal works in the intended direction but deserves more dedicated tests.
- Cable keyed mode as a trunk-level concept has been removed from the active model.
- Bus/cable bridge visuals are still evolving.
- Save/reopen, undo/redo, grouping/ungrouping, and clipboard behavior need more manual testing around bridge routes.
- Save/reopen and undo/redo need more manual testing around cable redraw, especially redraw-to-delete and redraw-to-new-endpoint.
- Collapse behavior needs more test coverage.
- Route drawing around vertical-to-vertical bridge cases may still need visual review.
- Generated app target derivation for complex cable/bus graphs needs dedicated tests.

## Future Directions Discussed

Possible `TRUNK` terminology:

- Internally rename the connection class from `BUS` to `TRUNK` once cable behavior stabilizes.
- Meaning: `TRUNK` is any loose/shared routeable trunk segment used to facilitate connections.
- Concrete types remain `Bus` and `Cable`.
- This should be delayed because it touches route strings, serialization assumptions, undo paths, and many existing names.

User-configurable keyed mode:

- Add explicit user control only if users need to override receiver selectivity defaults.
- Non-selective receiver: accepts all compatible messages.
- Selective receiver: accepts compatible messages only when name/alias matches.
- The default can remain UI-driven: bus input connections selective, cable input connections non-selective.

Visual direction:

- Cable tacks:
  - bullets when non-selective,
  - bullets with an outline when selective.
- Bridge tacks:
  - logically have both a cable tack and a bus tack,
  - visually draw only one clear tack at the bridge point,
  - avoid bus arrows because bridge flow is bidirectional at the topology level.
- Cable and bus line styles should remain visually distinct.

Model direction:

- Keep endpoint tacks as structural cable endpoint markers unless this causes real model complexity.
- Avoid adding explicit `from` / `to` cable properties unless endpoint tacks prove insufficient.
- Continue deriving endpoint/trunk relationships from tack arrays and flags.

Testing direction:

- Add unit tests for cable-to-bus bridge rxtx derivation.
- Add tests for keyed cable filtering through a bus.
- Add tests for duplicate bridge rejection.
- Add tests for save/reopen of bridge tacks and endpoint flags.
- Add tests for collapse after route cancellation and disconnection.
- Add tests for route-to-route promotion while drawing.
- Add tests for cable redraw with discarded tacks, reconnect to pin/pad, and release-without-endpoint deletion.
